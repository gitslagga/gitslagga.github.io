像高级工程师一样使用 Git

发表于 2022-12-03 更新于 2025-06-06 分类于 DevTools

Git 是一个强大的工具，当您知道如何使用它时，使用起来感觉很棒。

多年来，我已经在多个团队和项目中使用 Git 的这些功能。我仍在围绕一些工作流程（比如压缩或不压缩）发表意见，但核心工具功能强大且灵活（并且可编写脚本！）。

浏览 Git 日志

Git 日志是开箱即用的。

git 日志是基本的

使用git log给你一些信息。但它的信息量很大，通常不是您想要的。

让我们变得真实。这些日志没有给任何人留下深刻印象。他们很无聊。它们充满了你现在并不真正需要的信息。您正试图对项目中发生的事情有一个高层次的理解。

有更好的方法。

具有更多可见性的 git 日志

使用--graph并且--format我们可以快速获得项目中 git 提交的摘要视图。

哇！这些是一些好看的日志！旁边甚至还有一棵分枝的树。

这些日志向您展示了谁一直在处理什么、何时进行了更改以及您的更改在哪里适合更大的图景。

--graph将树形图添加到左侧。它不是最时尚的图表，但它有助于可视化项目分支中的变化。（在这里阅读文档。）

--format让您自定义日志的格式。有预设格式可供选择，或者您可以像本示例一样编写自己的格式。（在这里阅读文档。）

--all包括日志中的所有参考、标签和分支（包括远程分支）。您可能不想要_所有内容_，因此请根据需要进行调整。（在这里阅读文档。）

有关如何升级 git 日志的更多信息，请参阅 git-log 文档。→

了解特定的提交

您经常想了解特定提交发生了什么。git show可以向您显示提交中更改的高级视图，但它也可以让您查看对特定文件的更改。

查看提交的摘要

使用该--stat标志，您将看到提交摘要以及更改的文件和有关更改方式的详细信息。

查看提交的特定文件更改

当您想深入了解特定文件中的特定行更改时，请使用git show文件路径。

这为您的文件提供了特定的行更改。默认情况下，它将向您显示行更改以及两端的 3 行附加行，以提供更改行在文件中位置的上下文。

有关如何提升对 git commit 的理解的更多信息，请参阅 git-show 文档。→

进行更改

您已经在项目上创建了一个分支，向您的分支提交了一些更改，并且您已准备好将这些更改合并回main. 自从您分支以来，另一位工程师对相同的文件进行了更改。😱

如果您使用 GitHub 之类的服务，您的 PR 会告诉您是否存在合并冲突。

将更改推送到main. 这很好，因为您不想破坏其他人所做的所有辛勤工作。

要开始在本地解决此问题，您通常会采用以下两种路径之一：merge或rebase.

git merge vs git rebase

当main您想要将分支上的更改合并到您的分支中时，您可以_合并_更改或从不同的点重新设置分支的_基线。_

merge从一个分支获取更改，并在一次_合并提交_中将它们合并到另一个分支。

rebase调整分支实际分支的点（即将分支从基础分支移动到新的起点）。

_通常_，当您想要上游分支的代码包含在您的分支时，您可以使用rebase命令。当您想要把您的分支代码放回上游分支时，您将使用 merge 命令。
